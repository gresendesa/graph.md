## Description

Normalize the project's own Scrum memory under `scrum/` so it consistently uses
the section metadata notation and local schema validation contract established
so far. The project memory should become a first-class example of MDBind's
section-backed and schema-backed documentation style.

## Scope

- Audit the current `scrum/` memory files and identify where section metadata
  should be introduced or normalized.
- Define local JSON Schema files for the main memory record types where useful,
  including backlog item records, sprint records, decision records,
  architecture records, and experience records.
- Add or normalize section metadata blocks in project memory files without
  deleting historical content.
- Add `schema` references using the B-036 file-relative schema resolution
  contract.
- Add intelligent `@include` and `@ref` links where they create useful project
  memory connections, such as linking sprint records to backlog items,
  decisions to affected architecture records, and incidents to prevention
  records.
- Preserve consolidator files as synthetic summaries according to the
  constitution.
- Validate the memory with `mdb validate --root .`.
- Update architecture, decisions, backlog, sprint, and experience memory when
  relevant.

## Out of scope

- Changing the constitution unless the PO explicitly requests it.
- Rewriting historical records for style only when they can be preserved as
  legacy content.
- Implementing web URI schema references from B-034.
- Changing public CLI contracts.
- Committing before explicit PO acceptance.

## Risk assessment

- Implementation risk: medium, because existing memory files are mostly

- Contract risk: medium, because the memory format becomes a stronger example

  rewritten destructively.
- Validation risk: medium, because `mdb validate --root .` may surface
  unrelated legacy inconsistencies.

## Notes

- The PO assigned Priority 2 (high) during SPR-2026-17 planning.
- This item intentionally dogfoods the schema validation contract.
- The first implementation pass should prefer narrow, practical schemas over
  schemas that try to model every historical nuance.
- Smart links should distinguish `@include` from `@ref`: use `@include` only
  when composition should embed source content, and `@ref` when the memory
  should record a navigational or dependency relationship.
- Smart links should avoid reciprocal edges unless a cycle is intentionally
  useful for navigation.
