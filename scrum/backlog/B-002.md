## Descricao

Implementar o pipeline de parsing que transforma texto Markdown bruto em objetos `ParsedSection`
validados, cobrindo as etapas 1 a 3 do pipeline da spec (section 2).

Pipeline: `Markdown → AST → RawSection → ParsedSection`

## Escopo

### Etapa 1 - Geracao de AST
- Integrar `markdown-it-py` para tokenizacao do texto Markdown
- Produzir lista plana de tokens

### Etapa 2 - Section Discovery (RawSection)
- Varrer tokens buscando `heading_open`
- Ao encontrar, registrar `token_start`, `source_start_line`
- Continuar ate fim do documento ou proximo `heading_open` de nivel <= atual
- Registrar `token_end`, `source_end_line`

### Etapa 3 - Metadata Binding (ParsedSection)
- Dentro dos limites da RawSection, buscar bloco de codigo marcado como `section`
- Parsear conteudo YAML do bloco
- Validacoes obrigatorias (spec section 3):
  - Bloco `section` deve ser o primeiro bloco textual apos o heading (erro: "payload nao e o primeiro bloco")
  - Apenas um bloco `section` por RawSection (erro: "bloco section duplicado")
  - Campo `id` obrigatorio (erro: "secao sem payload obrigatorio")
  - Par `(arquivo, id)` deve ser unico no arquivo

## Criterios de aceitacao

- [x] Arquivo .md com 3 secoes aninhadas gera 3 objetos RawSection com limites corretos
- [x] Secao sem bloco `section` gera ParsedSection com metadata vazio e sem erro
- [x] Secao com bloco `section` sem `id` levanta erro de validacao
- [x] Secao com bloco `section` apos texto normal levanta erro de validacao
- [x] Dois blocos `section` na mesma secao levantam erro de validacao
- [x] `source_start_line` e `source_end_line` correspondem exatamente as linhas no arquivo fonte
- [x] Testes em `tests/test_parser.py` com fixtures .md cobrindo os casos acima (14/14)

## Riscos

- Medio. A logica de delimitacao hierarquica de headings pode ter edge cases com headings de nivel
  alternado (ex: h2 dentro de h1, seguido de outro h1).
