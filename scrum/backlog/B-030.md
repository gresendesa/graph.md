# B-030 - Structured query pseudo-fields and regex predicates

- Status: done
- PO Priority: 3 (medium)
- Risk: 3 (high)
- Owner: gresendesa
- Creation date: 2026-06-09
- Last updated: 2026-06-11
- Dependencies: B-023

## Description

Improve `mdb query` so agents and users can filter repository sections by
structural attributes without depending on helper tags such as `tag:backlog`.
Keep `mdb search` simple; the advanced predicate complexity belongs to
`mdb query`.

Today, `mdb query` supports metadata predicates such as `key=value`,
substring predicates such as `key~=value`, `tag:value`, and boolean operators.
That is enough for:

```bash
.venv/bin/mdb query "tag:backlog AND NOT status=done" --root scrum/backlog --json
```

However, the command cannot directly query structural fields such as section
identifier, file path, or heading, and it does not support regex predicates.
This limits queries such as selecting only root backlog item sections matching
`backlog.item.B-XXX`.

## Scope

- Expose structural pseudo-fields in `mdb query`, including `section`, `id`,
  `path`, `file`, and `heading`.
- Add regex matching syntax for query predicates, for example
  `key~=/regex/`.
- Preserve existing `key~=text` substring behavior.
- Ensure boolean combinations with regex predicates continue to work with
  `AND`, `OR`, and `NOT`.
- Preserve existing JSON output shape unless a versioned contract change is
  explicitly approved.
- Do not extend `mdb search` with this advanced query complexity in this item.
- Document the new query syntax in `README.md`, `specification.md`, and
  `llms.txt`.

## Example queries

```bash
mdb query "section~=/^backlog\\.item\\.B-\\d{3}$/ AND NOT status=done" --root scrum/ --json
mdb query "id~=/^backlog\\.item\\.B-\\d{3}$/ AND NOT status=done" --root scrum/ --json
mdb query "path~=scrum/backlog/ AND section~=/^backlog\\.item\\.B-\\d{3}$/ AND NOT status=done" --root . --json
mdb query "section~=/^sprint\\.SPR-\\d{4}-\\d{2}$/ AND status=review"
```

## Implementation notes

- `src/mdbind/cli.py` now evaluates `mdb query` predicates against the full
  `ParsedSection`, not only `metadata`.
- Structural pseudo-fields resolve as follows: `section` and `id` use the
  section identifier, `path` and `file` use `section.file_path`, and `heading`
  uses the Markdown heading text.
- `key~=/regex/` performs Python regex matching against the resolved field
  value.
- Existing `key~=text` substring matching remains case-insensitive and
  unchanged.
- Invalid regex syntax returns a clear query parse error with exit code 1.

## Validation

- Targeted tests:
  `.venv/bin/python -m pytest tests/test_cli_ia_v2.py` passed: 37 passed.
- Full regression:
  `.venv/bin/python -m pytest` passed: 216 passed.
- Manual query validation:
  `.venv/bin/mdb query "section~=/^conceito-.*/ AND path~=conceitos.md" --root tests/fixtures/repo --json`
  returned `conceito-a` and `conceito-b`.
- Manual structural field validation:
  `.venv/bin/mdb query "heading~=Conceito AND file~=conceitos.md" --root tests/fixtures/repo --json`
  returned `conceito-a` and `conceito-b`.
- Manual invalid regex validation:
  `.venv/bin/mdb query "id~=/[/" --root tests/fixtures/repo --json`
  exited with code 1 and reported `invalid regex`.

## Dependencies

## Risk assessment

- Parser risk: high, because `key~=/regex/` extends the current query grammar
  and must not break existing substring predicates.
- Contract risk: medium, because adding pseudo-fields should be additive, but
  the output and matching semantics must stay stable.
- Test risk: medium, because the change needs fixtures that distinguish
  root sections from nested subsections.
